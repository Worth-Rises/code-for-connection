1. Overview
What We're Building
A comprehensive admin portal for the Code for Connection platform — a corrections facility communication management system. The portal serves facility administrators and agency administrators who oversee all communications between incarcerated persons and their families, visitors, and legal contacts.
Scope Decisions
Decision
	Choice
	Ownership
	Everything under /admin — dashboard, all monitoring views, all management screens
	Target maturity
	MVP for pilot deployment; production-grade gaps tracked in TODO.md
	Contact model
	Keep FamilyMember for communications; add separate Visitor model with its own approval workflow
	Search capability
	Full-text search + configurable keyword alerts (vector/semantic search deferred to production)
	Change tracking
	Audit log for admin actions + entity version history for key entities
	RBAC
	Keep 2 roles (facility_admin, agency_admin); add feature-level permission flags
	Priority
	Comprehensive spec; all sections equal priority; build incrementally
	Industry Context
Research into corrections technology platforms (Securus/THREADS, ViaPath/Command, ICSolutions, JPay) reveals these as table-stakes features for any admin portal:


* Content moderation queues with claimed-item coordination
* Contact approval workflows with attorney-client privilege protection
* Real-time call monitoring with terminate capability
* Keyword alert systems for automated flagging
* Audit trails for every admin action
* Role-scoped access (facility admins can only see their facility)
* Shift handoff awareness ("since your last login" summaries)

* Facility selector: Dropdown showing facilities the admin has access to. Agency admins see all + "All Facilities" option. Facility admins see only their facility (selector hidden or disabled).
* Notifications bell: Count of items needing attention (pending contacts + pending messages + pending visitors + flagged content). Clicking opens a dropdown with grouped action items.
* Admin name dropdown: Profile, logout.
Design Principles
1. Task-oriented primary nav — organized by what staff need to do, not database entities
2. Badge counts on nav items — pending items shown as red badges (e.g., Contacts shows "7" for pending approvals)
3. "Since your last login" panel — top of Dashboard, shows deltas since the admin's last session
4. Confirmation dialogs for destructive actions — Deny, Terminate, Block, Remove all require explicit confirmation with reason field
5. Facility-scoped by default — facility admins see only their facility's data; agency admins can filter or view aggregate

5. RBAC & Permissions
Roles
Role
	Scope
	Description
	agency_admin
	All facilities in agency
	Full access across all facilities. Can switch facility context. Sees aggregate views.
	facility_admin
	One facility
	Access scoped to their assigned facility only.
	Permission Flags
Both roles can have individual permissions toggled on/off via AdminPermission. Default: all permissions granted on first creation.


Permission
	Controls
	manage_contacts
	Approve/deny/remove contacts, flag attorney privilege
	monitor_calls
	View active calls, call history, call detail, terminate calls
	review_messages
	View message queue, approve/block messages, view conversations
	manage_visitors
	Review visitor applications, approve/deny/suspend visitors
	manage_housing
	Move residents between units, edit unit type rules
	manage_settings
	Edit facility config, manage blocked numbers
	view_audit_log
	View audit log and entity history
	manage_blocked_numbers
	Add/remove blocked numbers
	run_reports
	Access reports and export data
	manage_keyword_alerts
	Create/edit/deactivate keyword alerts
	view_flagged_content
	View and triage flagged content queue

9. Cross-Cutting Concerns
Full-Text Search Implementation
Use PostgreSQL's built-in full-text search (tsvector + tsquery):


* Add tsvector generated columns on: incarcerated_persons, family_members, visitors, messages
* Create GIN indexes for fast lookup
* Search endpoint combines results across tables with relevance scoring
* No external search engine needed for MVP
Keyword Alert Processing
Messages flow:


1. Message created with status: pending_review
2. Background job scans message body against active KeywordAlert entries
3. If match found → create FlaggedContent entry linking to the message
4. Message appears in review queue with flag indicator


Call metadata flow (MVP — no audio transcription):


1. Call completes → metadata available
2. Scan contact phone number and call notes against keyword alerts
3. Flag if matched
Real-Time Updates
Active call lists need auto-refresh. Two options:


* MVP (polling): Frontend polls active call endpoints every 15-30 seconds via usePolling hook. Simple, works now.
* Production TODO: WebSocket subscription via existing Socket.io signaling server for push-based updates.
Audit Logging
Server-side middleware pattern:


// Helper used in every mutation route handler


async function auditLog(adminId: string, action: AuditAction, entityType: string, entityId: string, details?: any, ipAddress?: string) {


  await prisma.auditLog.create({ data: { adminUserId: adminId, action, entityType, entityId, details, ipAddress } });


}


Every POST/PATCH/DELETE handler calls auditLog() after the mutation succeeds.
Entity History Tracking
For key entities (residents, contacts, visitors, housing assignments), use a helper:


async function trackChange(entityType: string, entityId: string, fieldName: string, oldValue: any, newValue: any, changedBy: string) {


  await prisma.entityHistory.create({ data: { entityType, entityId, fieldName, oldValue: JSON.stringify(oldValue), newValue: JSON.stringify(newValue), changedBy } });


}


Called before updating the entity, capturing the old value for comparison.
Facility Scoping
Every admin API endpoint applies facility scoping:


// Agency admins: optional facilityId filter (null = all facilities)


// Facility admins: forced to their facilityId


function getFacilityFilter(user: AuthUser, queryFacilityId?: string) {


  if (user.role === 'agency_admin') return queryFacilityId ? { facilityId: queryFacilityId } : {};


  return { facilityId: user.facilityId };


}
Error Handling
Uses existing createSuccessResponse / createErrorResponse pattern from @openconnect/shared.
